2/15/25

After migrating computers, had an issue pulling down the Neocities repo, but another repo worked just fine. I had previously worked in Ubuntu, but VS Code is supposed to function for Windows, so it couldn't simply be that I was running on the wrong OS.

Error messages included Unable to read file: ENOENT: no such file or directory and Clone succeeded, but checkout failed. Some of these messages were explicable, like the one that told me that I wasn't working in a git repo. Others, less so.

The output at least identified one file that was causing a problem. Some Googling suggested that the problem had to do with file size, which...was odd, but I downloaded a few other things and tried some configurations meant to alleviate this. No dice.

Going back to the file identified in the output, I decided to shorten the name and see what happened. It worked! There was a new file identified as a problem, but that was alright, I was going to keep going at this. After I edited the second file name, everything was fine and I cloned my repo.

Then, trying to rename the two problem files, I was told that I was typing an invalid name.

It turns out that I was running on the wrong OS, as it turns out. Linux allows colons in file names, but Windows doesn't. Shortening the file names fixed the problem completely by accident: not because the file names were somehow too long but because they had colons in them.

Upshot: I need to be aware of the subtle differences between operating systems, because something that works fine in one situation may completely fall apart in another because of a single character.
